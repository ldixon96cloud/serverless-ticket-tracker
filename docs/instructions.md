# Configuration Summary

This project is a serverless REST API deployed on AWS using the Serverless Application Model (SAM), consisting of five Python 3.12 Lambda functions, a single DynamoDB table, and an API Gateway HTTP API — all defined and provisioned as infrastructure-as-code in `infra/template.yaml`. Each Lambda function is scoped to one operation (create, get, list, update, delete) and is wired directly to a specific HTTP route on the API Gateway, so an inbound HTTP request — for example `POST /tickets` — triggers only the `CreateTicketFunction` with no shared execution environment between functions. Every function receives the DynamoDB table name at runtime through a `TABLE_NAME` environment variable injected by SAM globals, and connects to DynamoDB using the `boto3` AWS SDK for Python. The table itself uses a single partition key (`ticketId`, a UUID string) with on-demand billing, meaning no capacity planning is required — DynamoDB scales automatically with request volume. IAM access is least-privilege by design: read-only functions (`GetTicketFunction`, `ListTicketsFunction`) are granted `DynamoDBReadPolicy`, while write functions are granted `DynamoDBCrudPolicy` — both scoped exclusively to the `Tickets` table. Conditional writes are used on update and delete operations so that DynamoDB rejects requests for non-existent tickets at the database level, returning a `ConditionalCheckFailedException` that the Lambda handler catches and converts into a clean `404` response.

# User-Facing Workflow Summary

From a user's perspective, interacting with this API follows a natural support ticket lifecycle. A user opens a ticket by sending a `POST /tickets` request with a title, description, priority, and the channel where the issue occurred (such as POS, KIOSK, DMB, or DELIVERY). The API immediately responds with the full ticket record including a system-generated unique ID, a status of `OPEN`, and timestamps — confirming the ticket has been recorded. At any point the user can retrieve the full list of active tickets with `GET /tickets`, optionally narrowing results by passing a `status` or `channel` query parameter (for example `?status=OPEN&channel=POS`) to focus on what's relevant to them. When a support agent picks up a ticket, they use `PATCH /tickets/{ticketId}` with `{"status": "IN_PROGRESS"}` to signal work has begun, and once resolved they send the same request with `{"status": "RESOLVED"}` — the API updates only the changed fields and refreshes the `updatedAt` timestamp, returning the full updated record so the caller always has the latest state. If a ticket was created by mistake or is a duplicate, it can be permanently removed with `DELETE /tickets/{ticketId}`, which returns a confirmation that the record was deleted. Throughout all of this, the API enforces data integrity silently in the background — invalid status values, unrecognized channels, and requests targeting tickets that don't exist are all rejected with clear error messages and appropriate HTTP status codes, so users never have to guess why a request failed.
